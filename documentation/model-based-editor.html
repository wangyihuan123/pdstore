Project Requirements: Model-Based Editor

This document captures the requirements for all of Gerald's model-based editor projects. All these editors have key features in common. They do not only allow you to edit a document in a most comfortable way: In fact they allow you to do this together with others on remote computers and to see the edits of the other person immediately! Have a look at the slides. The following list of requirements contains some longer and some very short explanations or just keywords (groups, layers) and will of course be discussed in more detail at project commencement.

Requirements common for all Model-Based Editor Projects

  • For a single user, the tool has two main views, called the history pane and the artifact pane.
  • The artifact pane is the pane where the user makes the ordinary edits of the document.
  • The history pane gives a logical view of the changes in the document so far and is used to perform the novel higher-level edits. The history pane is similar for all the model-based editors
  • If the user makes edits in the artifact pane, the logical representation of the edit appears in the history pane.
  • The simplest view of the history pane shows a list. The user can select a part of the artifact view and only sees changes relevant to the selection. This should also work for the current field of view in place of the marked section.
  • The content of both panes is remotely updated in a best-effort manner. If another collaborator makes an edit, it appears in the user's history pane and artifact pane, provided that it is in the user's field of view.
  • The history pane allows us to perform higher order operations: User operations can be rearranged.
  • In particular, the history pane allows us to perform counterfactual edits, by adding operations back in time: If the artifact view is set back to a certain point in time, then operations can be inserted. If one then goes forward to a later point in time, then later operations are applied to the result of the new operation.
  • Both panes supports logical tags. Both panes support a nice blame feature.
  • The user edits have to have suitable rubberbands, animations and stills associated with them. They are used to show updates by the collaborators as well as to replay the document history. Replaying the history allows the user to jump to places where edits have been made.
  • There must be a copy operation. If both copies were edited, the copy operation can be moved to later, resulting in a merge.
  • On copied objects, the counterfactual edit that applies the changes directly to the original can be performed as an option.

Technical requirements

  • The tool has to follow a strict model-view pattern.
  • Both panes are views of an internal model. The tool stores its model in the PDStore lightweight database system. Due to this technology this model is automatically synchronized, no further programming is required for this. The views however have to be updated with the observer pattern.
  • Uninterruptible user interface: No popups like errors, warnings - they are provided as a status field.

Diagram Editor

  • Double Buffering
  • Groups
  • Layers
  • Affine Transformations
  • Snap to x- and y-tabstops.
  • Replace graphical basic shapes late (not through counterfactual history)

Nice possible examples for elegant solutions

  • The palette pane in a graphical editor is just another document.
  • Layers are expressed through higher-order edits.
  • The clipboard is just another document.

Semantic Text editor

This is instant messenger meets wiki talk.
  • Text can be marked, history is only shown for that part. Local undo as simple application of general higher edits.
  • Between lines, a syntax tree can be unfolded.
  • Syntax highlighting based on the syntax tree.
  • The syntax tree can be manually edited, can support later automatic generation.
  • Graphical Support for tabs.
  • Nice search and replace, copied with higher-level edits: Selected replacements are counterfactually replaced with copy/pastes.
  • Other important known features: making special characters visible, convert to uppercase,

Tree Editor

There will be some nice novel presentation features, but another focus is on synchronizing with and acting on an outside system, e.g. a file system.
  • recognition and update of external changes. They are also included into the history! Means some reverse engineering.
  • Execution of changes in the outside world. Check for intended effect.
  • Planning stage: Actions can be done "in the future", are not yet executed.
  • Timed execution of planned operations.
  • Path view at the top.
  • Neighbor frame: If a parent is about to scroll out of the top, it remains visible. These remaining parents form part of the path representation on top. If the direct lower neighbor of a parent is scrolling out of the bottom it remains visible as well.
  • Presentation improvements for trees. Vertical lines in the tree view enable fast scrolling.

Deliverables

  • The code as an eclipse project, submitted through the version control system. All submitted code must be under a free license.
  • Part of the projects should be unit tests of all important test cases.
